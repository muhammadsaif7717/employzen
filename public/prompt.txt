These rules MUST be followed for every single component generated.
No exceptions.

RULE 1 — FILE TYPE
  Every component is a .tsx file (TypeScript + JSX).
  File name: PascalCase → e.g. JobCard.tsx, CandidateSidebar.tsx

RULE 2 — TYPESCRIPT INTERFACE (REQUIRED IN EVERY FILE)
  Every component MUST define a TypeScript interface for its props
  at the top of the file, before the component function.
  Example:
    interface JobCardProps {
      id: string
      title: string
      company: string
      location: string
      salary: string
      type: 'full-time' | 'part-time' | 'remote' | 'internship' | 'contract'
      status: 'pending' | 'approved' | 'rejected'
      postedAt: string
      onClick?: () => void
    }
  If there are no props, still write: interface ComponentNameProps {}

RULE 3 — STYLING: TAILWIND ONLY IN className=""
  ALLOWED   : className="bg-blue-600 text-white px-4 py-2 rounded-lg"
  ALLOWED   : className={cn("base-class", condition && "extra-class")}
  FORBIDDEN : style={{ color: '#fff', padding: '8px' }}
  FORBIDDEN : CSS Modules (.module.css)
  FORBIDDEN : styled-components or emotion
  FORBIDDEN : Hardcoded hex colors anywhere in the file
  ALL styling must be done through Tailwind utility classes in className="".

RULE 4 — SHADCN/UI USAGE
  Use shadcn/ui components as the BASE structure/behavior layer.
  Override their appearance using the className prop with Tailwind.
  Common shadcn imports:
    import { Button }       from "@/components/ui/button"
    import { Input }        from "@/components/ui/input"
    import { Badge }        from "@/components/ui/badge"
    import { Card, CardContent, CardHeader, CardTitle } from "@/components/ui/card"
    import { Dialog, DialogContent, DialogHeader, DialogTitle } from "@/components/ui/dialog"
    import { Select, SelectContent, SelectItem, SelectTrigger, SelectValue } from "@/components/ui/select"
    import { Tabs, TabsContent, TabsList, TabsTrigger } from "@/components/ui/tabs"
    import { Avatar, AvatarFallback, AvatarImage } from "@/components/ui/avatar"
    import { Separator } from "@/components/ui/separator"
    import { Table, TableBody, TableCell, TableHead, TableHeader, TableRow } from "@/components/ui/table"
    import { DropdownMenu, DropdownMenuContent, DropdownMenuItem, DropdownMenuTrigger } from "@/components/ui/dropdown-menu"
    import { Sheet, SheetContent, SheetTrigger } from "@/components/ui/sheet"
    import { Textarea } from "@/components/ui/textarea"
    import { Label } from "@/components/ui/label"
    import { Checkbox } from "@/components/ui/checkbox"

  Override example:

      Apply Now
    </Button>

RULE 5 — cn() UTILITY (class merging)
  Always use cn() from shadcn for conditional/merged classes:
    import { cn } from "@/lib/utils"
    className={cn("base", isActive && "active-class", variant === 'primary' && "primary-class")}

RULE 6 — DARK MODE
  Every component must support dark mode.
  Use Tailwind dark: prefix on every color class.
  dark: classes are auto-activated via the "class" strategy in tailwind.config.ts.

RULE 7 — AI AUTO-DETECTION FOR UNKNOWN/EXTRA COMPONENTS
  If asked to build a component NOT listed in the known components
  section of this file, the AI MUST:
  (a) Identify the component PURPOSE and CONTEXT (which user role
      sees it, what action it represents)
  (b) Map it to the nearest semantic color from this theme:
      User action / CTA        → blue-600 (primary)
      Success / completion     → emerald-600
      Warning / pending        → amber-600
      Error / delete / reject  → red-600
      Candidate context        → blue palette
      Employer context         → teal palette
      Admin context            → purple palette
      Neutral / info display   → slate palette
      Premium / highlight      → violet palette
  (c) Use the card, input, badge, and button patterns from this file
  (d) NEVER introduce a new color family not listed in this theme
  (e) ALWAYS include a TypeScript interface and dark mode support

RULE 8 — Responsive Design Requirement
   All components must be fully responsive across every screen size and device type, including mobile, tablet, laptop, desktop, and large displays. Use Tailwind CSS responsive utilities to ensure layouts adapt seamlessly without breaking, overlapping, or causing horizontal scrolling at any viewport width.

RULE 9 — UI/UX Quality Standard
  The user interface must be clean, modern, and visually consistent, with strong attention to spacing, alignment, typography, and color harmony. The user experience should be intuitive, smooth, and user-friendly, ensuring easy navigation, clear interaction flow, and accessibility across all pages.

RULE 10 — Production-Level Code Quality Standard
   All code must be written at a production-ready standard, following clean code principles and industry best practices. Functions should be modular, reusable, and well-structured with a single responsibility. Code must be readable, maintainable, and scalable, with meaningful naming conventions and consistent patterns. Add clear, concise comments only where necessary to explain complex logic—not obvious code. Avoid redundancy, ensure proper error handling, and write code as a professional developer would in a real-world production system.

RULE 11 — STATE MANAGEMENT DISCIPLINE
    Local state (useState) only for UI-specific logic
    Shared state must be minimal and intentional
    Avoid unnecessary global state
    Keep state as close as possible to where it is used
    No over-engineering of state architecture

RULE 12 — API LAYER SEPARATION
    All API calls MUST be placed in /services or /lib/api
    Components MUST NOT contain direct fetch/axios logic (unless trivial demo)
    API functions must be reusable and typed
    Always return consistent response structures
    Handle loading, error, success explicitly

RULE 13 — ERROR HANDLING STANDARD
    Never ignore errors in async operations
    Always implement try/catch for API calls
    Show user-friendly error messages (not raw system errors)
    Provide fallback UI for failed states
    Prevent application crashes under all conditions

RULE 14 — SCALABILITY REQUIREMENT
    Every component must be reusable by design
    Avoid hardcoded values; use props instead
    Structure components for future feature expansion
    Keep business logic separate from UI logic
    Design with multi-role system in mind (Candidate / Employer / Admin)

RULE 15 — UX FEEDBACK LOOP (CRITICAL)
    Every user action MUST have feedback:
    Loading state during async actions
    Success confirmation on completion
    Error feedback on failure
    Buttons must show disabled state when processing
    Avoid silent UI updates

RULE 16 — CODE CONSISTENCY RULE
    Maintain strict naming conventions:
    Components → PascalCase
    Functions → camelCase
    Constants → UPPER_CASE
    File names MUST match component names
    Keep folder structure consistent across modules
    Do not mix architectural patterns

RULE 17 — ACCESSIBILITY ENHANCEMENT (A11Y+)
    All interactive elements must be keyboard accessible
    Ensure focus states are visible and clear
    Use proper aria attributes when needed
    Maintain minimum contrast ratio for readability
    Avoid UI elements that rely only on color meaning

RULE 18 — LOGIC SIMPLICITY RULE
    Prefer simple logic over complex abstractions
    Avoid premature optimization
    If logic becomes complex, extract into utility functions
    Keep components readable at first glance
    Favor clarity over clever code

RULE 19 — DEBUGGING FRIENDLINESS
    Code must be easy to debug and trace
    Use meaningful variable names (no abbreviations like x, tmp)
    Avoid deeply nested logic
    Keep side effects predictable
    Log only in development mode if necessary

RULE 20 — FINAL SYSTEM INTEGRITY RULE
    Every component must:
    Compile without warnings
    Be production-safe
    Follow all previous rules strictly
    No rule can be skipped even for "small components"
    Consistency > shortcuts

When asked to build any component — whether listed in Section 22
or completely new/extra — the AI MUST follow these steps:

STEP 1 — IDENTIFY CONTEXT (which user role uses this?)
  Candidate context  → blue palette as primary
  Employer context   → teal palette as primary
  Admin context      → purple palette as primary
  Public / shared    → blue palette as primary

STEP 2 — IDENTIFY PURPOSE (what does this component do?)
  Primary CTA / submit / apply      → blue-600 buttons + blue accents
  Data display / read-only info     → white card + slate text (neutral)
  Success / completion state        → emerald accents
  Warning / attention / pending     → amber accents
  Error / destructive / rejection   → red accents
  Premium / featured / highlighted  → violet accents
  Secondary info / tips             → sky accents

STEP 3 — APPLY STRUCTURE (mandatory for ALL components)
  (a) TypeScript interface at top of .tsx file
  (b) shadcn/ui primitives as base (Button, Card, Input, Badge, etc.)

ThemeToggle.tsx component:
  "use client"
  import { useTheme } from 'next-themes'
  import { Sun, Moon } from 'lucide-react'
  import { Button } from '@/components/ui/button'

  interface ThemeToggleProps {}

  export default function ThemeToggle({}: ThemeToggleProps) {
    const { theme, setTheme } = useTheme()
    return (
      <Button variant="ghost" size="icon"
        className="text-slate-500 hover:text-blue-600 hover:bg-blue-50
                   dark:text-slate-400 dark:hover:text-blue-400 dark:hover:bg-blue-950
                   rounded-lg relative"
        onClick={() => setTheme(theme === 'dark' ? 'light' : 'dark')}>
        <Sun className="h-5 w-5 rotate-0 scale-100 transition-all dark:-rotate-90 dark:scale-0" />
        <Moon className="absolute h-5 w-5 rotate-90 scale-0 transition-all dark:rotate-0 dark:scale-100" />
      </Button>
    )
  }
